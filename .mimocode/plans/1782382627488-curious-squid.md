# Plan: Image Support Across All Translators

## Problem

Image content blocks are silently dropped during Anthropic↔OpenAI translation. The current code only handles `text`, `tool_use`, and `tool_result` content types. Images in the request content array are simply ignored, meaning any request with images loses that content.

Additionally, `sanitize_openai_payload` incorrectly flattens messages containing only image blocks (no text) into an empty string, destroying image data.

## Image Format Reference

**Anthropic format:**
```json
{"type": "image", "source": {"type": "base64", "media_type": "image/jpeg", "data": "<base64>"}}
```

**OpenAI format:**
```json
{"type": "image_url", "image_url": {"url": "data:image/jpeg;base64,<base64>"}}
```

## Files to Modify

### 1. `core/providers/translation.py` (primary translation logic)

**`anthropic_to_openai_request`** (lines ~95-166):
- Add `image` block handling alongside `text` and `tool_use` in the list-content loop
- Convert Anthropic `{"type": "image", "source": {"type": "base64", "media_type": "...", "data": "..."}}` to OpenAI `{"type": "image_url", "image_url": {"url": "data:<media_type>;base64,<data>"}}`
- When message has image blocks (no text), set content to the image block list (don't flatten to string)

**`openai_to_anthropic_request`** (lines ~206-257):
- Add `image_url` block handling in the content processing
- Convert OpenAI `{"type": "image_url", "image_url": {"url": "data:<media_type>;base64,<data>"}}` to Anthropic `{"type": "image", "source": {"type": "base64", "media_type": "...", "data": "..."}}`
- Handle mixed content arrays (text + images)

**`sanitize_openai_payload`** (lines ~13-64):
- Update the flattening logic: only flatten to string when ALL parts are text-type or strings
- Preserve `image_url` blocks in `non_text_parts` so they pass through correctly
- Currently image blocks would be put in `non_text_parts` which is correct, but the fallback logic needs review

### 2. `translator.py` (legacy translation, used by unit tests)

Mirror the exact same changes as `core/providers/translation.py`. These two files contain duplicate translation logic.

### 3. `core/router.py` (token estimation)

**`_estimate_request_tokens`** (lines ~331-345):
- Handle `image_url` content type in the content iteration loop (currently only handles `text`)

### 4. `tests/test_proxy.py` (unit tests)

Add test functions:
- `test_anthropic_to_openai_request_with_image`: Anthropic request with image block → verify OpenAI output has `image_url` with correct data URI
- `test_openai_to_anthropic_request_with_image`: OpenAI request with `image_url` → verify Anthropic output has `image` source block
- `test_mixed_content_with_images`: Messages containing both text and images in same content array
- `test_sanitize_preserves_images`: Verify `sanitize_openai_payload` doesn't strip image blocks

### 5. `load_test/mock_server.py` (integration test mock)

Add `/v1/messages` endpoint (Anthropic format) that:
- Parses incoming request body
- Detects image content blocks in messages (accepts but doesn't process them)
- Returns a standard text response (no image output, per user requirement)

The mock server currently only handles `/v1/chat/completions`. Adding `/v1/messages` enables Anthropic-format integration tests.

### 6. `load_test/load_test.py` (integration test client)

Add a test mode or option that sends requests with image content blocks to exercise the image translation path through the proxy.
